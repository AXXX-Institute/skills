The three skills — and the order to use them

Start with the quick-start, scaffold your repo, then keep the reference handy.

StepSkillRoleWhat it does — and when to reach for it
1 · Start here mlspace-jobs-quick-start SETUP First time on MLSpace? Run /mlspace-jobs-quick-start — interactive, one-step-at-a-time setup: conda env → mls install → credentials → your first submitted job. Explicit-only.
2 · Scaffold your repo mlspace-jobs-scaffold BUILD In your real training repo, run mlspace-jobs-scaffold: generates run_train_jobs.py, run_eval.py, and experiments.py implementing the five pillars and discovering your instance types. Done on a green --dry run — no GPU spent. After that, launching a sweep is editing one file and running one command.
3 · Reference mlspace-jobs OPERATE Keep it handy for day-to-day operation and when something breaks: commands (status / logs / wait / kill), accelerate multi-GPU config, and troubleshooting recipes & known bugs (CUDA/image segfaults, /dev/shm "no space left").

The five pillars — and why each matters

Every scaffolded launcher reproduces these. Each exists to kill a specific failure mode seen on real MLSpace runs.

1Experiments as code

Every run is an ExperimentConfig in experiments.py, collected by collect_experiments(); launchers iterate the registry and never hard-code a run.
Why it matters: a sweep you can't diff is a sweep you can't reproduce. Encoding runs as data makes the whole batch inspectable and reviewable in one place — the foundation the other four build on.

2Idempotent launches

Before submitting, check a terminal marker the entrypoint writes last; if it's there, skip — unless --force.
Why it matters: re-running after a partial batch should submit only what's missing, never duplicate completed work or clobber good outputs. The check is a cheap exists().

3In-progress dedup

List Pending/Running jobs via mls and skip submitting when a normalized job description already matches.
Why it matters: idempotency only catches finished work. A queued job has written no artifacts yet — without this, a second launch wastes your allocation on a duplicate.

4Out-of-workdir artifacts

Write to absolute paths on shared NFS, with a separate root for sanity/CI runs.
Why it matters: jobs run in ephemeral containers and the launch host may be a CI checkout that gets wiped. Writing under the repo loses outputs or bloats git; absolute NFS paths stay durable and reachable from every node.

5Code staging

Stage the repo from .git into a per-commit directory and point the job there; refuse to submit a dirty tree.
Why it matters: a job may start minutes after you submit it while you keep editing. If it read your live checkout, an unrelated edit could silently change what a running experiment executes — irreproducible and dangerous.
