The three skills — and the order to use them

Start with the quick-start, scaffold your repo, then keep the reference handy.

SkillRoleWhat it does — and when to reach for it
mlspace-jobs-quick-start SETUP First time on MLSpace? Run /mlspace-jobs-quick-start — interactive, one-step-at-a-time setup: conda env → mls install → credentials → your first submitted job. Explicit-only.
mlspace-jobs-scaffold BUILD In your real training repo, run mlspace-jobs-scaffold: generates run_train_jobs.py, run_eval.py, and experiments.py implementing the properties below and discovering your instance types. Done on a green --dry run — no GPU spent. After that, launching a sweep is editing one file and running one command.
mlspace-jobs OPERATE Keep it handy for day-to-day operation and when something breaks: commands (status / logs / wait / kill), accelerate multi-GPU config, and troubleshooting recipes & known bugs (CUDA/image segfaults, /dev/shm "no space left").

Why scaffold experiments as code

mlspace-jobs-scaffold builds launchers with four properties baked in. Each card shows what you get — hit Without on the right to see what breaks if you skip it.

1Experiments as code

Every run is an ExperimentConfig row in experiments.py, collected by collect_experiments() — so a whole sweep is one expression.

With

The whole sweep is one expression — diffable, reviewable, reproducible.

EXPERIMENTS = [
    ExperimentConfig(name=f"sft-{m}-lr{lr}", model=m, lr=lr)
    for m in ("qwen3-8b", "llama3-8b")
    for lr in (1e-5, 2e-5, 5e-5)
]
Without

One YAML per run, kept in sync by hand:

  • A sweep becomes a pile of near-identical files to create and edit
  • No single place to diff or review what's actually running
  • Copy-paste drift — one file keeps a stale hyperparameter
  • Reproducing the batch means reconstructing it from memory

2Idempotent launches

One command launches the whole sweep and is safe to re-run: finished runs skip (a terminal marker), and jobs already Pending/Running skip too (in-progress dedup).

With

One command, re-run as often as you like — it submits only what's missing. No bookkeeping.

# re-run any time — only missing experiments go out
python run_train_jobs.py
#   finished        → skipped (marker)
#   queued/running  → skipped (dedup)
Without

You track launch state by hand:

  • Re-running resubmits duplicates that burn your GPU allocation
  • …or clobbers good outputs by re-running completed work
  • After a partial failure you hand-pick exactly what to retry
  • Two people launching the "same" sweep double-submit it

3Code staging

At submit time the launcher snapshots your code from .git into a per-commit directory and points the job there.

With

The job runs a frozen snapshot of the exact commit you submitted — whenever it starts, however the working tree changes.

# submit now — the job runs THIS commit, even days later
<STAGING_ROOT>/<commit_hash>/    # frozen snapshot; the job cd's in
Without

The job reads your live checkout, and a queued job may start hours or days later:

  • Edits made after submitting silently change what runs
  • Switching branches repoints a pending job at the wrong code
  • Two jobs from the "same" script can execute different code
  • No record of what actually ran — impossible to reproduce

4Out-of-workdir artifacts

Outputs — and the per-commit code snapshots from the previous property — land at absolute paths on shared NFS, outside any repo checkout or git worktree.

With

Artifacts and staged code live outside every worktree, so cleanups can't touch them; all worktrees, branches, and commits share one durable tree.

/home/jovyan/<proj>-artifacts/
    code/<commit>/        # staged snapshots (property 3)
    training/<exp>/       # checkpoints, metrics, eval
# survives git clean / worktree remove / branch switch
Without

Artifacts under the repo are fragile — especially with git worktrees and agents:

  • An agent running git clean -fdx to "tidy up" wipes your checkpoints
  • git worktree remove takes that worktree's artifacts with it
  • Each worktree writes its own copy → scattered, colliding results
  • Outputs bloat the repo or get committed by accident
