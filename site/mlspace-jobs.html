The three skills — and the order to use them

Start with the quick-start, scaffold your repo, then keep the reference handy.

StepSkillRoleWhat it does — and when to reach for it
1 · Start here mlspace-jobs-quick-start SETUP First time on MLSpace? Run /mlspace-jobs-quick-start — interactive, one-step-at-a-time setup: conda env → mls install → credentials → your first submitted job. Explicit-only.
2 · Scaffold your repo mlspace-jobs-scaffold BUILD In your real training repo, run mlspace-jobs-scaffold: generates run_train_jobs.py, run_eval.py, and experiments.py implementing the properties below and discovering your instance types. Done on a green --dry run — no GPU spent. After that, launching a sweep is editing one file and running one command.
3 · Reference mlspace-jobs OPERATE Keep it handy for day-to-day operation and when something breaks: commands (status / logs / wait / kill), accelerate multi-GPU config, and troubleshooting recipes & known bugs (CUDA/image segfaults, /dev/shm "no space left").

What every launcher gives you — with vs. without

Four properties the scaffold bakes in. For each: what your life looks like without it, and with it.

1Experiments as code

Every run is an ExperimentConfig row in experiments.py, collected by collect_experiments() — so a whole sweep is one expression.

Without

One YAML per run, kept in sync by hand. A sweep means creating — and later editing — a pile of near-identical files.

configs/sft-lr1e5-qwen.yaml
configs/sft-lr2e5-qwen.yaml
configs/sft-lr5e5-qwen.yaml
configs/sft-lr1e5-llama.yaml
configs/…      # and on, by hand
With

The whole sweep is one line — diffable, reviewable, reproducible.

EXPERIMENTS = [ExperimentConfig(name=f"sft-{m}-lr{lr}", model=m, lr=lr)
               for m in ("qwen3-8b", "llama3-8b") for lr in (1e-5, 2e-5, 5e-5)]

2Idempotent launches

One command launches the whole sweep and is safe to re-run any time: finished runs are skipped (a terminal marker), and jobs already Pending/Running are skipped too (in-progress dedup) so you never submit a duplicate.

Without

You track which runs finished and which are still queued, then hand-pick what to resubmit — or risk duplicate jobs that burn your GPU allocation.

With

One command, re-run as often as you like — it submits only what's missing. No bookkeeping.

# re-run any time — only missing experiments go out
python run_train_jobs.py
# finished  -> skipped (marker)
# queued/running -> skipped (dedup)

3Out-of-workdir artifacts

Outputs land at absolute paths on shared NFS — outside any repo checkout or git worktree.

Without

Artifacts under the repo are fragile around git worktrees: a stray cleanup — say an agent running git clean -fdx or removing a worktree to "tidy up" — silently deletes your checkpoints.

git worktree remove ../exp-2   # takes its artifacts with it
git clean -fdx                 # wipes results under the repo
With

Artifacts live outside every worktree, so cleanups can't touch them — and all worktrees, branches, and commits share one durable tree.

/home/jovyan/<proj>-artifacts/training/…  # absolute, on NFS
# survives git clean / worktree remove / branch switch

4Code staging

At submit time the launcher snapshots your code from .git into a per-commit directory and points the job at it.

Without

A queued job can start hours or days later. If it reads your live checkout, whatever you edited — or the branch you switched to — meanwhile is what actually runs, not the code you submitted.

With

The job runs a frozen snapshot of the exact commit you submitted — whenever it starts, however the working tree changes.

# submit now; runs THIS commit even if it starts days later
<STAGING_ROOT>/<commit_hash>/   # frozen snapshot the job cd's into
