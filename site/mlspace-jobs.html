Why "experiments as code"?

A sweep scattered across shell flags, notebooks, and ad-hoc mls invocations can't be reviewed, diffed, or reproduced. Two weeks later nobody can say which flags produced which checkpoint — and re-running the batch means reconstructing it from memory.

The mlspace-jobs-scaffold skill builds your launchers around one idea: every experiment is a row in experiments.py — an ExperimentConfig dataclass instance, gathered by a collect_experiments() registry. The launcher imports that registry and iterates it. Adding or removing an experiment is a diff to one file, reviewed like any other code change.

That registry is also the single per-project home of the instance-type map, so training and evaluation agree on hardware. Everything else the launcher does — idempotency, dedup, code staging — operates over that registry. Get the registry right and the rest follows.

The five pillars — and why each matters

Every scaffolded launcher reproduces these. Each exists to kill a specific failure mode seen on real MLSpace runs.

1Experiments as code

Every run is an ExperimentConfig in experiments.py, collected by collect_experiments(); launchers iterate the registry and never hard-code a run.
Why it matters: a sweep you can't diff is a sweep you can't reproduce. Encoding runs as data makes the whole batch inspectable and reviewable in one place — the foundation the other four build on.

2Idempotent launches

Before submitting, check a terminal marker the entrypoint writes last; if it's there, skip — unless --force.
Why it matters: re-running after a partial batch should submit only what's missing, never duplicate completed work or clobber good outputs. The check is a cheap exists().

3In-progress dedup

List Pending/Running jobs via mls and skip submitting when a normalized job description already matches.
Why it matters: idempotency only catches finished work. A queued job has written no artifacts yet — without this, a second launch wastes your allocation on a duplicate.

4Out-of-workdir artifacts

Write to absolute paths on shared NFS, with a separate root for sanity/CI runs.
Why it matters: jobs run in ephemeral containers and the launch host may be a CI checkout that gets wiped. Writing under the repo loses outputs or bloats git; absolute NFS paths stay durable and reachable from every node.

5Code staging

Stage the repo from .git into a per-commit directory and point the job there; refuse to submit a dirty tree.
Why it matters: a job may start minutes after you submit it while you keep editing. If it read your live checkout, an unrelated edit could silently change what a running experiment executes — irreproducible and dangerous.

The three skills

SkillRoleWhat it does
mlspace-jobs-scaffold BUILD Generates run_train_jobs.py, run_eval.py, and experiments.py implementing the five pillars, discovering your allocation's instance types along the way. Done when both launchers print correct payloads on a green --dry run — no GPU spent.
mlspace-jobs-quick-start SETUP Interactive, one-step-at-a-time first-time setup: conda env → mls install → credentials → your first submitted job. Explicit-only — launch it with /mlspace-jobs-quick-start.
mlspace-jobs OPERATE The reference skill: day-to-day commands (status / logs / wait / kill), accelerate multi-GPU config, and troubleshooting recipes & known bugs (segfaults, shared-memory errors).

Recommended workflow

Start with the quick-start guide

First time on MLSpace? Run /mlspace-jobs-quick-start. It walks you through creating a conda environment, installing and configuring the mls CLI, and submitting + monitoring a first job end-to-end.

Run the scaffold in your real project repo

In your actual training repo, ask to run mlspace-jobs-scaffold. It reads your repo, discovers your allocation's instance types, and writes experiments.py plus the launchers — ending on a green --dry run. From then on, launching a sweep is editing one file and running one command.

Keep it running with the reference skill

mlspace-jobs is just a reference: reach for it for day-to-day operation and when something breaks — troubleshooting recipes and known bugs (CUDA/image segfaults, /dev/shm "no space left", accelerate multi-GPU config).
